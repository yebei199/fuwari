起因很朴素：有个搜索服务静默挂了五天没人发现。容器是 `restart: unless-stopped`，被手动停过一次后这条策略就把"停止"记住了，daemon 重启也不会拉起，于是它一直 Exited 躺着，对外只表现为网关 502。

我想要一个命令，一次性扫过**所有**服务、一个都不漏、告诉我哪些没在跑。听起来是十分钟的活，结果第一版直接跑挂，值得记一笔。

## 第一版：复用 `verify`，逻辑对、完全不可用

编排器里早就有 `service verify <name>`：解析服务 → 建传输 → 按服务类型（compose / 代理 / k8s）跑对应的存活校验。"扫全部"看起来就是把它套个循环：

```rust
for svc in ctx.config.services.values() {
    match verify_one(&ctx, svc).await {
        Ok(())  => Status::Running,
        Err(e)  => Status::Down(e.to_string()),
    }
}
```

迭代的是配置里同一份服务表（和 `service list` 用的一样），所以"不漏任何一个"这点是结构性保证的——这部分思路没问题。逻辑也对：单测全过，校验流程完全复用，零重复代码。按"复用已有能力"这条懒人准则，这是教科书式的第一选择。

然后真机一跑，`timeout 300` 把进程杀了，**一行输出都没有**。

## 两个本质问题

复盘下来，"复用单项 verb 做批量"踩中了两个一眼看不出来的坑：

**一是扇出成本。** `verify_one` 每调一次就**新建一条到目标主机的 SSH 连接**。本地到主力机要过一层 SOCKS5 代理，握手本身就一两秒；二十多个服务串行下来就是二十多次握手。更糟的是 k8s 服务的 verify 会做 `kubectl rollout status` 这种带等待的检查，单个就能阻塞好几秒。25 个服务 × 十几秒，轻松破 300 秒。单项校验里无所谓的"建连一次"，在批量里成了主导项。

**二是输出缓冲。** 我把结果收集进 `Vec`、最后 `render_report` 一次性打印。单服务校验时这没区别；但在一个跑几分钟的扫描里，这等于"全部跑完才出字"。一旦中途超时被杀，**前面所有已经查出来的结果全丢了**——你甚至不知道它卡在第几个。长任务"先攒后印"是反模式。

这两点合起来，命令在逻辑正确的同时彻底不可用。复用是对的直觉，但用错了粒度。

## 第二版：按主机批处理 + 流式输出

把粒度从"每服务一次远程往返"换成"每主机一次远程往返"：

- **每台 Docker 主机只跑一次 `docker ps -a --format '{{.Names}}\t{{.State}}'`**，拿回全主机容器状态快照，再在内存里逐个比对期望容器。一台主机十几个容器，从十几次 inspect 压成一次。
- **每个 k8s 集群只跑一次 `kubectl get pods --all-namespaces`**，扫出所有非 `Running`/`Completed` 的 pod。
- 期望容器的来源用配置里 compose stack **已声明的健康检查项**做唯一真值——部署系统认定该跑的，就是 check 该查的，两边同源，不另搞一份清单。

整个扫描的远程调用从"服务数（25+）"降到"主机数 + 集群数"（个位数）。同时改成**边查边打印**：每个单元一出结果立刻 `println!` 一行，最后再补一行汇总。这样进程被打断也保留已有进度，慢扫描全程有反馈。

```text
  OK    infrastructure/postgres
  DOWN  general/searxng                exited
  OK    proxy
  DOWN  k8s/local-k8s:kube-system/nvidia-device-plugin-...  CrashLoopBackOff
  SKIP  cloudflare-security            (no container)

22 running, 1 down, 1 skipped
```

重写后全量扫描**约十秒**出全表，准确标出那个 Exited 的搜索容器、另一个被强杀的服务，以及一个本机集群里 CrashLoop 的 GPU 插件 pod；有任何 down 就 `exit 1`，能直接进 CI 或定时任务。

## 几条可复用的结论

- **"复用现成的 verb"是批量操作的第一直觉，也常常是错的。** 当单项操作里有"每次新建连接"这类隐藏的固定成本时，循环 N 次会把这个常数项放大成主导项。批量要先想"能不能一次往返覆盖一批"，而不是"把单次套个 for"。
- **长扫描必须流式输出。** "先攒后印"在会超时、会被打断的场景下等于把已完成的工作也一起丢掉。一出结果就打印，是几乎零成本的健壮性。
- **期望集合要从配置真值推导，而不是手写。** 用部署系统已声明的健康检查项当唯一来源，"不漏任何一个"才是结构性的，而不是靠记性。
- **什么时候反过来才对：** 如果单项检查是本地的、廉价的（无网络、无握手），那直接复用单项 verb 循环完全没问题——别为不存在的扇出成本提前优化。坑只在"每项都贵"时才出现。

最后留一个细节：这次扫描里那个 `restart: unless-stopped` 容器静默躺了五天，恰恰说明"重启策略"不等于"存活监控"。策略只保证进程崩了自己爬起来，但对"被人为停掉后该不该拉起"这类语义无能为力——这正是需要一个外部 check 命令兜底的地方。
