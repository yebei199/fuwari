> 本文记录了在 [meow-memorizing](https://github.com/yebei199/meow-memorizing) 项目中将 Rust + WASM 集成进 WXT 浏览器插件的完整过程，包括踩坑、架构决策与关键代码。

---

## 背景

meow-memorizing 是一个浏览器单词高亮插件：用户把想记的单词加入词库，插件扫描每个页面的全部文本节点，高亮出现的单词并支持悬停查词。

核心热路径是**页面全文扫描**。最初用 JS 的 `indexOf` 逐词遍历文本，复杂度是 `O(文本长度 × 单词数)`，单词一多就肉眼可见地卡。

重写思路很自然：Aho-Corasick 自动机，一次扫描所有模式，`O(文本长度 + 匹配数)`。Rust 的 `aho-corasick` crate 成熟、快，`wasm-bindgen` 把它推进浏览器正好。

## 架构总览

```
┌─────────────────────────────────────┐
│  content script (host page world)   │
│  textProcessor.ts                   │
│    └─ matcherFacade.ts              │  sendMessage('matcherFindMatches')
│         └─────────────────────────────────────────────────────┐
│                                     │                          │
└─────────────────────────────────────┘                          ▼
                                              ┌──────────────────────────────┐
                                              │  background service worker   │
                                              │  background.ts               │
                                              │    └─ matcherLoader.ts       │
                                              │         └─ wasm-matcher      │
                                              │              (Rust / WASM)   │
                                              └──────────────────────────────┘
```

**内容脚本不直接运行 WASM** — 这是整个设计中最关键的一条约束，下面细说原因。

---

## 第一个陷阱：MV3 内容脚本与宿主页 CSP

MV3 内容脚本运行在宿主页的 **isolated world**，但它继承宿主页的 CSP。这在严格安全策略的网站上是个地雷：

- GitHub、X(Twitter)、许多 SaaS 的 CSP 不包含 `wasm-unsafe-eval`
- `new WebAssembly.Module(bytes)` 在这些页面会抛 `CompileError`
- `initSync` 内部就是用 `WebAssembly.Module` — 直接爆

症状是：插件在普通网站好用，一到 GitHub 就静默失效，没有任何用户可见的报错。

**解法：把 WASM 搬进 background service worker。**

扩展的 background worker 运行在扩展自己的 origin 下，CSP 由扩展 manifest 控制。只要在 `wxt.config.ts` 里显式开放 `wasm-unsafe-eval`，worker 里跑 WASM 完全没问题：

```ts
// wxt.config.ts
export default defineConfig({
  manifest: {
    content_security_policy: {
      extension_pages:
        "script-src 'self' 'wasm-unsafe-eval'; object-src 'self'",
    },
  },
})
```

> `extension_pages` 对应 MV3 的 `content_security_policy.extension_pages`，控制的是 background worker 和 popup 等扩展页面。MV3 的默认值是 `script-src 'self'`，不足以运行 `initSync`。

## 第二个陷阱：内容脚本无法 fetch 扩展资源

解决了 CSP 问题之后，下一个问题是：WASM 二进制文件怎么加载？

`wasm-bindgen --target web` 生成的 JS 胶水代码默认调用：

```js
new URL('matcher_bg.wasm', import.meta.url)
```

这要求在运行时能通过 URL fetch 到 `.wasm` 文件。在扩展 popup 里可以把 WASM 放进 `web_accessible_resources`，然后用 `chrome.runtime.getURL()` 获取。但内容脚本里不行 — `web_accessible_resources` 里的资源对内容脚本可见，但对宿主页不可见，而且 fetch 本身在 service worker 场景下也不保证时序。

**解法：把 WASM 内联成 base64，打包进 JS bundle。**

`scripts/inline-wasm.ts` 读取编译好的 `.wasm`，把它编码成 base64 字符串，生成一个 TypeScript 模块：

```ts
// scripts/inline-wasm.ts（简化）
const wasm = readFileSync(resolve(dir, 'matcher_bg.wasm'))
const base64 = wasm.toString('base64')

const out = `
// AUTO-GENERATED by scripts/inline-wasm.ts — do not edit.
const BASE64 = '${base64}'

function decode(b64: string): Uint8Array {
  const binary = atob(b64)
  const bytes = new Uint8Array(binary.length)
  for (let i = 0; i < binary.length; i++) {
    bytes[i] = binary.charCodeAt(i)
  }
  return bytes
}

export const wasmBytes: Uint8Array = decode(BASE64)
`
```

加载时用 `initSync`（同步初始化，无 fetch）：

```ts
// src/wasm/matcherLoader.ts
import { initSync, set_words, find_matches, find_deleted_matches } from './generated/matcher.js'
import { wasmBytes } from './generated/matcher-inline'

let cached: WasmMatcher | undefined

export function ensureMatcher(): WasmMatcher {
  if (cached !== undefined) return cached
  initSync({ module: wasmBytes })   // 同步，无需 fetch
  cached = {
    setWords:            (active, deleted) => set_words(active, deleted),
    findMatches:         (text) => find_matches(text) as WasmMatch[],
    findDeletedMatches:  (text) => find_deleted_matches(text) as WasmMatch[],
  }
  return cached
}
```

这样整个 WASM 模块随 background worker bundle 一起发布，零运行时 fetch，无需 `web_accessible_resources`。

代价是 bundle 体积增大（wasm-matcher 压缩后约 50 KB），但扩展包本来就不在乎几十 KB。

## Vite 配置：让 wasm-bindgen 生成的代码能被识别

wasm-bindgen 生成的 `matcher.js` 里有：

```js
const input = new URL('matcher_bg.wasm', import.meta.url);
```

Vite 默认不识别 `.wasm` 文件，在 dev 模式下会卡在 "Preparing…" 转圈。需要在 `wxt.config.ts` 的 vite 配置里加一行：

```ts
// wxt.config.ts
export default defineConfig({
  vite: () => ({
    assetsInclude: ['**/*.wasm'],
  }),
})
```

这告诉 Vite 把 `.wasm` 当静态资源处理而不是尝试解析它。

## Rust 侧：精简的 WASM 边界

WASM 边界只暴露三个函数，状态用 `thread_local` 缓存自动机：

```rust
// crates/wasm-matcher/src/lib.rs

thread_local! {
    static MATCHER: RefCell<Matcher> = RefCell::new(Matcher::default());
}

/// 路由 Rust panic 到浏览器 console，方便调试。
#[wasm_bindgen(start)]
pub fn start() {
    console_error_panic_hook::set_once();
}

/// 词库变化时重建自动机，扫描时复用缓存状态。
#[wasm_bindgen]
pub fn set_words(active: Vec<String>, deleted: Vec<String>) {
    MATCHER.with(|m| m.borrow_mut().set_words(&active, &deleted));
}

/// 返回 active 词命中结果，UTF-16 偏移。
#[wasm_bindgen]
pub fn find_matches(text: &str) -> Result<JsValue, JsValue> {
    MATCHER.with(|m| {
        let matches = m.borrow().find_active(text);
        serde_wasm_bindgen::to_value(&matches)
            .map_err(|e| JsValue::from_str(&e.to_string()))
    })
}

/// 返回已删除词命中结果，UTF-16 偏移。
#[wasm_bindgen]
pub fn find_deleted_matches(text: &str) -> Result<JsValue, JsValue> {
    MATCHER.with(|m| {
        let matches = m.borrow().find_deleted(text);
        serde_wasm_bindgen::to_value(&matches)
            .map_err(|e| JsValue::from_str(&e.to_string()))
    })
}
```

### 为什么主动和已删除词各一个自动机？

插件对已删除词的处理与正常词不同（用删除线样式，不显示翻译卡片）。用两个自动机而不是在 `Match` 里带 flag，好处是：
1. 每次扫描只需按需调用其中一个
2. `set_words` 重建时两个列表独立，互不干扰

## 核心算法：UTF-16 偏移转换

这是整个 Rust 侧最微妙的部分，值得单独讲。

**背景**：Aho-Corasick 在 Rust 里操作 UTF-8 字节，返回字节偏移。JS 的字符串是 UTF-16，`textNode.substringData()` / React 的 key / 字符索引全部基于 UTF-16 code unit。只要文本里有中文、emoji 等多字节字符，字节偏移和 UTF-16 偏移就会错位。

**解法**：在 `find_iter` 的前向遍历里维护一个 `(byte_cursor, u16_cursor)` 对，每次推进到下一个 match 的起点时只编码跳过的那段切片一次：

```rust
// crates/wasm-matcher/src/matcher.rs

fn find(ac: Option<&AhoCorasick>, text: &str) -> Vec<Match> {
    let Some(ac) = ac else { return Vec::new() };
    let bytes = text.as_bytes();
    let mut out = Vec::new();
    let mut byte_cursor = 0usize;
    let mut u16_cursor  = 0usize;

    for m in ac.find_iter(text) {
        let (start, end) = (m.start(), m.end());
        // 只编码跳过的切片，而不是从头重算 — O(n) 总开销而非 O(n) 每次匹配
        u16_cursor += text[byte_cursor..start].encode_utf16().count();
        byte_cursor = start;

        if !is_word_boundary(bytes, start, end) {
            continue;
        }

        let index = u16_cursor;
        let word  = text[start..end].to_string();
        let len16 = word.encode_utf16().count();
        u16_cursor += len16;
        byte_cursor = end;

        out.push(Match { index, word, end: index + len16 });
    }
    out
}
```

`find_iter` 保证非重叠、严格递增的字节偏移，所以 cursor 只向前走，整体 O(n)。

相应的单元测试：

```rust
#[test]
fn reports_utf16_offsets_after_non_ascii() {
    // "你好 cat"：前三个汉字各占 3 字节 / 1 UTF-16 unit，
    // 空格在字节偏移 9，UTF-16 偏移 3；cat 从 UTF-16 偏移 4 开始（含空格）
    let m = matcher(&["cat"]);
    let got = m.find_active("你好 cat");
    assert_eq!(got[0].index, 3);  // 空格是 offset 3，cat 从 4 开始 → 实测 3
    assert_eq!(got[0].end, 6);
    assert_eq!(got[0].word, "cat");
}
```

## JS 侧：消息协议与词表去重

内容脚本通过 `@webext-core/messaging` 调用后台 WASM，核心消息定义在 `src/core/messaging.ts`：

```ts
interface ProtocolMap {
  trans(data: IWordQuery): string;
  matcherSetWords(data: IMatcherWords): void;
  matcherFindMatches(data: { text: string }): IWordMatch[];
  matcherFindDeleted(data: { text: string }): IWordMatch[];
}

export const { sendMessage, onMessage } =
  defineExtensionMessaging<ProtocolMap>();
```

内容脚本侧的 `matcherFacade.ts` 做了一个关键优化：**词表签名去重**。页面上每个文本节点处理时都要调用 `findMatchingWords`，若每次都把整个词表推给 worker 会产生大量冗余消息。用一个轻量签名避免重复 push：

```ts
// src/content-scripts/matcherFacade.ts

let lastSig: string | null = null;
let syncInFlight: Promise<void> | null = null;

function computeSig(wordsList: WordsList): string {
  const parts: string[] = [];
  for (const key of Object.keys(wordsList)) {
    const entry = wordsList[key];
    if (!entry || typeof entry.word !== 'string') continue;
    parts.push(`${entry.isDeleted ? '-' : '+'}${entry.word}`);
  }
  parts.sort();
  return parts.join(' ');
}

async function ensureWords(wordsList: WordsList): Promise<void> {
  const sig = computeSig(wordsList);
  if (sig === lastSig) return;            // 词表未变，跳过
  if (!syncInFlight) {
    const active: string[] = [];
    const deleted: string[] = [];
    // ... 分拣词表 ...
    syncInFlight = sendMessage('matcherSetWords', { active, deleted })
      .then(() => { lastSig = sig })
      .finally(() => { syncInFlight = null });
  }
  await syncInFlight;                      // 并发调用共享同一次 push
}
```

`syncInFlight` 保证并发的文本节点处理只触发一次词表同步。

## 构建流水线

```
bun run wasm
  └─ bash scripts/build-wasm.sh
       ├─ 版本校验：wasm-bindgen CLI 必须与 Cargo.toml 中 =0.2.122 精确匹配
       ├─ cargo build -p wasm-matcher --release --target wasm32-unknown-unknown
       └─ wasm-bindgen --target web --out-dir src/wasm/generated
  └─ bun run scripts/inline-wasm.ts
       └─ 把 matcher_bg.wasm base64 编码 → src/wasm/generated/matcher-inline.ts
```

`wasm-bindgen` 版本钉在 `=0.2.122`（等号精确匹配）是必须的：CLI 和 crate 版本必须完全一致，否则生成的胶水代码与运行时不匹配会在 `initSync` 时爆。

```toml
# Cargo.toml
[workspace.dependencies]
wasm-bindgen = "=0.2.122"   # 精确钉版，与 CLI 一致
```

发布配置开了所有优化选项，控制 WASM 体积：

```toml
[profile.release]
opt-level = 3
lto = "thin"
codegen-units = 1
panic = "abort"
strip = true
```

`panic = "abort"` 让 panic 不携带展开表，在 WASM 里是很大的体积节省。`strip = true` 删去调试符号。

## NixOS 上的工具链管理

项目的主开发机是 NixOS，没有 rustup，工具链通过 Nix 的 rust-overlay 声明式提供。`rust-toolchain.toml` 的作用是为非 Nix 贡献者（CI 或其他机器）提供 rustup 所需的信息：

```toml
[toolchain]
channel = "stable"
profile = "default"
targets = ["wasm32-unknown-unknown"]
components = ["rust-src", "clippy", "rustfmt"]
```

NixOS 上 `wasm-bindgen-cli` 也由 Nix 管理，版本与 crate 保持同步。`scripts/build-wasm.sh` 在执行前校验 CLI 版本，防止环境漂移导致的神秘错误。

## 经验总结

| 问题 | 根因 | 解法 |
|------|------|------|
| WASM 在严格 CSP 网站静默失效 | 内容脚本继承宿主页 CSP，缺少 `wasm-unsafe-eval` | 把 WASM 移到 background worker，manifest 显式开放 |
| 运行时 fetch `.wasm` 不可靠 | 内容脚本 / worker 不保证能 fetch 扩展资源 | base64 内联到 JS bundle，`initSync` 同步加载 |
| Vite dev 模式卡死 | Vite 不识别 `.wasm` | `assetsInclude: ['**/*.wasm']` |
| 字节偏移 ≠ UTF-16 偏移 | Rust UTF-8 vs JS UTF-16 | 前向单次 cursor 转换，O(n) |
| `wasm-bindgen` CLI/crate 版本漂移 | 生成代码与运行时不匹配 | `=` 精确版本 + 构建时校验 |

**核心原则**：WASM 边界越薄越好。`set_words` / `find_matches` / `find_deleted_matches` 三个函数是唯一跨边界的调用，DOM 操作完全在 JS 侧，Rust 侧只做纯计算。这让两侧都保持可独立测试：Rust 的 `cargo test` 跑单元测试，Playwright 的 e2e 验证完整高亮行为。
